## 1. 目标与背景

### 1.1 目标

- 构建一个 AI 驱动的知识图谱系统，将香港公开政策数据转化为可查询的动态洞察
- 实现实体准确率 >90%、关系覆盖率 >80% 的高质量知识图谱
- 通过 RAG 和 GraphRAG 技术实现多步推理，查询速度提升 2-5 倍
- 确保幻觉率 <5%，所有输出可追溯到源数据
- 短期发表学术论文，长期探索商业化路径

### 1.2 背景

EPIP（企业政策洞察平台）从 Policy-RAG 项目演进而来，专为处理香港 data.gov.hk 的异构数据集设计，包括 **24 个 CSV 数据文件** 和 **4 个 PDF 数据字典**（健康统计、预算、法规等）。PDF 数据字典包含字段定义、数据规范和元数据说明，是理解 CSV 数据语义的关键参考。当前政策数据分散在多个文件中，缺乏有效的关联查询和推理能力，决策者难以快速获取跨领域的政策洞察。

本平台采用 Light-RAG 框架构建知识图谱，结合手动优化确保数据质量，通过 ReAct 多步推理支持复杂查询（如"疫情政策 → 疫苗分发 → 医院资源"影响链分析）。本地部署架构确保数据隐私合规。

## 2. 需求

### 2.1 功能需求 (Functional Requirements)

#### 数据准备模块
- **FR1**: 系统应通过 Light-RAG 支持导入 CSV 文件和 PDF 文件
- **FR2**: 系统应自动验证数据格式完整性并报告错误
- **FR3**: 系统应支持数据预处理（缺失值处理、术语标准化）
- **FR4**: 系统应支持增量数据处理，避免重复导入

#### 知识图谱构建模块（Light-RAG 核心）
- **FR5**: 系统应使用 Light-RAG 从 CSV/PDF 自动构建知识图谱
- **FR6**: 系统应通过 Light-RAG 实现实体识别与嵌入生成
- **FR7**: 系统应通过 Light-RAG 实现实体消歧与去重
- **FR8**: 系统应通过 Light-RAG 实现关系提取与三元组生成
- **FR9**: 系统应通过 Light-RAG 实现子图融合与实体对齐
- **FR10**: 系统应支持 Light-RAG 参数调优（分块策略、置信度阈值等）
- **FR11**: 系统应提供手动优化界面，允许用户调整 Light-RAG 生成的实体和关系

#### 查询处理模块
- **FR12**: 系统应支持自然语言查询输入
- **FR13**: 系统应使用 LLM 解析查询，提取中心节点/实体
- **FR14**: 系统应支持 Cypher 图查询，使用 A*/Dijkstra 最短路径算法
- **FR15**: 系统应实现 ReAct 循环多步推理，支持分解 3-5 个子查询
- **FR16**: 系统应聚合多分支查询结果，基于置信度和路径长度排名
- **FR17**: 系统应缓存热门查询路径（Redis，TTL 1 小时）
- **FR18**: 系统应支持查询超时回退到基本路径查询

#### 幻觉检测与可追溯性模块
- **FR19**: 系统应对生成的回答进行多维度事实一致性验证
- **FR20**: 系统应为每个输出附加推理路径溯源信息
- **FR21**: 系统应记录结构化日志，包含步骤置信度和关键节点
- **FR22**: 系统应过滤置信度 <0.7 的弱推理点

#### 可视化模块
- **FR23**: 系统应使用 D3.js 展示推理树可视化
- **FR24**: 系统应支持 KG 嵌入路径可视化
- **FR25**: 系统应量化显示节点贡献权重（attention 机制）

#### 系统管理模块
- **FR26**: 系统应支持多租户隔离
- **FR27**: 系统应支持 RBAC 权限管理
- **FR28**: 系统应提供 Prometheus 监控指标

### 2.2 非功能需求 (Non-Functional Requirements)

#### 性能
- **NFR1**: 查询响应时间应比基线提升 2-5 倍
- **NFR2**: 系统应支持 TB 级知识图谱查询
- **NFR3**: 并发查询支持至少 10 个用户同时操作

#### 质量
- **NFR4**: 实体识别准确率应 >90%
- **NFR5**: 关系提取覆盖率应 >80%
- **NFR6**: 幻觉率应 <5%
- **NFR7**: 数据完整率应 >95%

#### 安全与隐私
- **NFR8**: 系统必须支持本地部署，数据不外传
- **NFR9**: 所有 API 访问需要认证
- **NFR10**: 敏感操作需要审计日志

#### 可维护性
- **NFR11**: 代码测试覆盖率应 >80%
- **NFR12**: 所有公共 API 需要文档
- **NFR13**: 支持 Docker 一键部署

#### 可扩展性
- **NFR14**: 模块化设计，支持替换 LLM 后端
- **NFR15**: 支持增量更新知识图谱

## 3. 技术假设

### 3.1 仓库结构
**Monorepo** - 单一仓库管理所有模块，便于协调开发和版本控制

### 3.2 服务架构
**模块化单体** - 初期采用单体架构，按功能模块划分，便于快速迭代；预留微服务拆分接口

### 3.3 测试要求
**单元测试 + 集成测试** - 核心算法需要单元测试，数据流需要集成测试，覆盖率目标 >80%

### 3.4 技术栈详情

| 层级 | 技术选择 | 理由 |
|------|----------|------|
| 语言 | Python 3.10+ | 丰富的 AI/ML 生态，Light-RAG 原生支持 |
| KG框架 | Light-RAG (HKUDS/LightRAG) | 轻量级 GraphRAG，支持 CSV/PDF 处理、图构建和检索 |
| 图数据库 | Neo4j | 成熟的图数据库，Cypher 查询语言，GDS 算法库 |
| 缓存 | Redis | 高性能缓存，支持 TTL |
| 向量嵌入 | sentence-transformers | 开源嵌入模型，本地部署 |
| 数据处理 | pandas | 数据预处理和清洗 |
| 可视化 | D3.js | 灵活的前端可视化库 |
| 部署 | Docker / Docker Compose | 标准化容器部署 |
| 监控 | Prometheus + Grafana | 开源监控方案 |

### 3.5 其他技术假设

- LLM 后端：初期使用本地 Ollama 或 API（如 OpenAI/Claude），需支持切换
- 数据集已下载到 `dataset/` 文件夹，包含：
  - **24 个 CSV 数据文件**：涵盖健康统计、住院统计、预算、食品安全等领域
  - **4 个 PDF 数据字典**：包含字段定义、数据规范和元数据说明
- 开发环境使用虚拟环境 (venv) 管理依赖
- 项目结构遵循 Python 标准布局 (src layout)
